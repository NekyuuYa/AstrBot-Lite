# 技术构想

## 分多状态机处理消息状态

 - 不活跃状态
 - review状态
 - 活跃状态（实时回复）
 - 半活跃（）

## 技术细则

1. 不再实时决策回复，采用不定时查询，或由ai决定查询历史消息时间
2. review模式可进行整合处理，将历史消息进行序号标记，以块状模式交给llm决策，要求其回复需要进行回复的序号，同时也利于上下文理解确实需要回复的情形
3. 多种转移方式可选用于进入活跃状态机，如
   - 短时单会话收到的消息数超过阈值
   - at消息
4. 参考注意力回落机制，设计合理的活跃转不活跃条件
5. 参考多agent设计，将`决策` `回复` `创建定时任务` `长期记忆总结` `上下文控制` `图片识别` `兴趣控制` `表情识别归类`多模块拆分给适合其功能的llm工作
6. 规范`回复`模块的prompt注入，设计规范的前缀以提高缓存命中率，在活跃状态机，可以考虑启用llm的显式缓存以降低成本
7. 对于回复prompt设计，参考方案如下
   - 将固定不变的prompt置于最前端，如`基础人格设定` `回复逻辑`等
   - 长期记忆紧随其后，这个部分降低更新频率
   - 随后是上下文，对于上下文，不要使用单条弹出的方式，而是选择批弹出，例如如果设置的上下文宽度上限是`50`条，那么在长度reach`50`时，一次弹出`5-10`条，可提高缓存利用效率
   - 上下文不一定是按顺序弹出，由特定的模型决定，在当前对话情形需要弹出的上文
   - 上下文主题分类，用于处理有时会出现的单会话多主题情况
   - review模式的上下文可包含部分下文
   - 上下文后放置一些易变性的设定，如情绪、回复倾向、好感度、着重标注的回复对象身份
   - 最后放置需要回复的目标消息
8. 对于长期记忆总结，这种对于失效要求不高的llm请求，可用批处理接口调用，以减低成本
9. 对于消息回复决策，回复的目标单位不应设计为`条`而应该是`块`
10. 注意数据库拆分和结构化，利于后续的迁移
11. 可选加入`会话主题转移`机制，将不同会话的主题迁移到其他会话，以实现话题的自启动
12. 合理的节律设计，为智能体设置作息，回复时限，回复兴趣水平，可选加入倦怠机制，使得回复的注意力下降
13. 根据人设控制主题兴味，在兴味不超过阈值时不响应该主题
14. 上下文弹出不一定要直接丢弃，可以通过compact的方式来控制长度
15. 注意向`回复`part的llm强调消息的线性排列关系
